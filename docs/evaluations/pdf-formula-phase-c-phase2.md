# PDF 实验性公式 Phase C：阶段 2 有限优化结果

日期：2026-07-20

## 裁决

三项限定优化均已实现并产生明显收益，但 **未达到预设可用停止线**。按约定停止继续堆启发式，进入阶段 3 方案选择；本报告中的阶段 3 路线均未实施。

## 改动

1. 同页邻近公式碎片先按二维位置重建为块，再经过原有 `0.6/0.65` 门禁。
2. 双栏行按所在栏宽和栏中心计算“居中短行”。
3. 公式编号作为块锚点，关联同栏上下片段；对 Attention 这类被 pdf.js 错分到左右栏、但处于同一视觉带且水平相接的片段做受限关联。
4. 编号加分必须同时存在操作符、数学符号或数学字体证据；避免把 BERT 的 `(1)(2)(3)` 枚举正文当公式。
5. 增加只读候选诊断：源行索引、源文本和块置信度；不写入 `PaperContent`，不改变 HTML/ar5iv 路径。

没有降低 `0.6/0.65` 阈值，没有加入 OCR、云服务或外部模型。

## 停止原因

验收前置标准要求残缺风险率降到 45% 以下；实际为 52%。虽然召回和核心样本显著改善，但继续调邻接半径、短 token 正则或编号规则只会在误报与漏检间搬动问题，无法恢复文本层已经丢失的二维语义。因此规则裁决为 **未完全达标**，停止继续增加启发式。

## 阶段 3 可选路线（均未实施）

### 1. arXiv TeX 源优先，PDF 块作为回退（推荐第一）

- 思路：检测到 arXiv ID 时下载公开 [source archive](https://arxiv.org/e-print/1706.03762)，安全解包但绝不执行 TeX；解析主文档、`\\input`、章节和 equation 环境，保留作者原始 LaTeX。失败或非 arXiv 再回退当前 PDF 块。
- 预期效果：对有 TeX 源的 arXiv 论文接近真 LaTeX，经典论文不再依赖 LLM 猜测；非 arXiv PDF 无收益。
- 代价/风险：多文件主入口、宏定义、条件编译、版本对应、压缩包安全、CORS/限流；复杂自定义宏仍需保守降级。
- 工作量：中高，约 5–8 个开发日，另需真实语料回归。
- 外部服务：依赖 arXiv 公共 source 下载，但不依赖 OCR、付费服务或上传第三方。

### 2. 本地公式图像 OCR companion（推荐第二，作为任意 PDF 回退）

- 思路：pdf.js 渲染候选区域高分辨率裁图，通过本机 Python/Docker/Native Messaging companion 调用 [pix2tex](https://github.com/lukas-blecher/LaTeX-OCR/blob/main/README.md) 类模型，再由 KaTeX 校验输出。
- 预期效果：覆盖任意可渲染 PDF，能绕过文本层字体名和二维结构丢失；复杂多行公式仍需校验。
- 代价/风险：模型体积、首次下载、CPU/GPU 时延、Windows/macOS/Linux 安装维护、区域检测误差；开源模型也明确要求人工复核。
- 工作量：高，约 1–2 周完成 companion、通信、安装与回归。
- 外部服务：无云服务；依赖本机运行时及模型文件。

### 3. 云端 STEM OCR API（效果优先但需明确授权）

- 思路：上传公式裁图或 PDF 到 [Mathpix OCR API](https://docs.mathpix.com/) 类服务，获取 LaTeX/Math Markdown，并保留当前本地路径作为离线回退。
- 预期效果：最快获得较高的任意 PDF 覆盖率，通常也能处理表格和整页数学内容。
- 代价/风险：论文内容外传、API Key、费用、速率限制、网络与供应商锁定；必须显式 opt-in、隐私提示和最小上传范围。
- 工作量：中，约 3–5 个开发日，另加隐私/配额/UI 工作。
- 外部服务：是，强依赖第三方云 API。

### 4. 基于 PDF glyph/operator 的本地数学版面解析（推荐第四）

- 思路：不再依赖 `getTextContent()` 的扁平行，读取 [pdf.js `getOperatorList()`](https://mozilla.github.io/pdf.js/api/draft/module-pdfjsLib-PDFPageProxy.html)、字形变换和字体对象，建立 glyph 空间图，再识别基线、上下标、分数线、根号和矩阵。
- 预期效果：对数字生成 PDF 比当前文本行方案更完整，也覆盖非 arXiv；扫描 PDF 仍无效，自定义字体和复杂结构仍有上限。
- 代价/风险：高度依赖 pdf.js 内部表示和字体映射，版本升级脆弱；等同于实现一个小型数学版面解析器。
- 工作量：很高，约 3–6 周并需更大标注集。
- 外部服务：无。

## 推荐顺序

1. **方案 1：arXiv TeX 源优先**——最符合 PaperLens 当前 arXiv 主场景，准确度/隐私/工作量比最好。
2. **方案 2：本地 OCR companion**——若必须覆盖任意 PDF，作为方案 1 的离线回退。
3. **方案 3：云 OCR**——仅在接受上传、费用和 Key 管理时，作为最快的效果路线。
4. **方案 4：glyph parser**——无外部依赖，但投入最大且技术上限仍低于源 LaTeX。

建议产品路线是“1 + 2”分阶段组合；当前等待用户选择，不执行。
